# 🧭 Campus Buddy — 校园活动搭子匹配图系统设计文档

本项目完全基于 AI Coding 结对编程模式开发，是一个无向无权异构图建模系统，用于从虚构的兴趣问卷和活动清单中构建社交网络图，帮助学生查询适合的校园活动、匹配志同道合的活动搭子，并分析兴趣社区的连通性。本系统同时提供了高性能的 Python 命令行 MVP 与现代化极客霓虹感（Slate & Neon）的 Vue 3 + D3.js + TypeScript 可视化 Web 界面。

---

## 📅 一、项目目标与最小功能范围

### 1. 项目目标
利用图论相关数据结构与搜索算法对校园社交关系进行建模，以兴趣标签作为“桥梁”介质，实现跨实体的双跳关系推荐。解决校园中“活动查找困难”和“孤立个体缺乏社交途径”的问题。

### 2. 最小功能范围 (MVP)
* **图的构建**：支持从 CSV 文件或增量 API 中读取学生、兴趣和活动数据，并建立对应的图关系。
* **邻接表保存**：利用邻接表（Adjacency List）保存图的拓扑结构。
* **推荐活动 (recommend_activities)**：从学生节点出发，基于“学生 -> 兴趣 -> 活动”两跳路径推荐活动。
* **推荐搭子 (recommend_buddies)**：从学生节点出发，基于“学生 -> 兴趣 -> 其他学生”两跳路径匹配拥有共同兴趣的搭子。
* **连通分量 (connected_components)**：识别图中所有互相独立的连通社群网络。
* **打印邻接表**：支持直接打印或返回图的邻接表表达。

## 📊 二、数据来源与数据表说明

为了支持系统分析，项目包含三类数据，物理上表现为三张 CSV 数据表，位于 `data/` 目录下：

### 1. 学生兴趣表 (`student_interests.csv`)
记录学生与兴趣标签的关联及兴趣强度。
* **字段**：`student` (学生姓名), `interest` (兴趣名称), `level` (兴趣强度，取值 1-5 整数)
* **示例**：
  ```csv
  student,interest,level
  小明,篮球,5
  小明,Python,4
  小明,摄影,2
  ```

### 2. 活动兴趣表 (`activity_interests.csv`)
记录校园活动与对应兴趣类型的分类关系，包含容量及时间。
* **字段**：`activity` (活动名称), `interest` (关联兴趣), `capacity` (活动容量), `time_slot` (时间段)
* **示例**：
  ```csv
  activity,interest,capacity,time_slot
  周三篮球赛,篮球,10,周三晚
  Python入门工作坊,Python,20,周四晚
  ```

### 3. 学生报名表 (`student_registrations.csv`)
记录学生已经报名的活动（作为直接关联边）。
* **字段**：`student` (学生姓名), `activity` (活动名称)
* **示例**：
  ```csv
  student,activity
  小明,周三篮球赛
  ```

## 🌐 三、图的拓扑结构设计

本项目采用**异构无向无权图 (Heterogeneous Undirected Unweighted Graph)** 对系统实体和关联进行建模。

### 1. 顶点类型 (Vertices)
图中的顶点分为三类，为了避免命名冲突，每个节点在图的键（Key）中均带有**类型前缀**：
1. **学生节点**：前缀为 `student:`（如 `student:小明`）
2. **兴趣节点**：前缀为 `interest:`（如 `interest:篮球`）
3. **活动节点**：前缀为 `activity:`（如 `activity:周三篮球赛`）

### 2. 边类型与语义 (Edges)
图中存在三类边，均为**无向、无权**：
* **学生 -- 兴趣 边**：语义为“某一学生对该项兴趣感兴趣”。
* **活动 -- 兴趣 边**：语义为“某一活动属于该项兴趣分类”。
* **学生 -- 活动 边**：语义为“某一学生已报名/注册参加该活动”（用于推荐去重）。

### 3. 设计决策分析

#### ① 为什么第一版使用无向无权图？
* **对称性**：学生感兴趣一个主题，等价于这个主题吸引了这个学生；活动和兴趣的归属也是双向的。如果使用有向图，两跳搜索时需要进行反向边处理，增加了算法复杂度。
* **无权图的简洁性**：在基础推荐 and 连通分量划分中，主要解决“有无关联”的结构性连通问题。第一阶段使用无向无权图可以保证 BFS 算法和邻接表遍历的运行效率，降低系统复杂度。在扩展功能中，我们通过独立的数据结构（如字典）来记录 `level` 和 `capacity`，既保持了核心图结构的简单性，又保留了扩展性。

#### ② 为什么邻接表（Adjacency List）适合本项目？
* **稀疏图特征**：校园社交网络中，顶点的度（学生感兴趣的主题数、活动的分类数）相比于最大可能连线数极为稀疏。邻接表空间复杂度为 $O(V + E)$，远小于邻接矩阵的 $O(V^2)$，极大地节省了内存空间。
* **高效邻居查询**：在进行两跳推荐（查找与当前节点相距 2 跳的节点）时，需要频繁迭代当前节点的所有邻居。邻接表能以 $O(\text{deg}(v))$ 的时间复杂度直接取出所有邻居，避免了邻接矩阵每一步都需要扫描整行（$O(V)$）的性能缺陷。

#### ③ 如何避免不同类型的节点重名？
若某项兴趣和某个活动都叫“篮球”，在没有前缀的情况下，它们会被合并为一个节点，破坏图的逻辑。
* **解决方案**：在图底层使用 `f"{kind}:{name}"` 拼接字符串作为节点的唯一标识符。即 `interest:篮球` 与 `activity:篮球` 在图中完全独立，有效防止命名空间污染。

## 🧠 四、核心算法设计与实现说明

### 1. 两跳活动推荐 (recommend_activities)
* **路径语义**：`Student -> Interest -> Activity`（2跳）。
* **算法步骤**：
  1. 获取学生节点 `student:A` 的所有邻居。
  2. 过滤出以 `interest:` 为前缀的节点（1跳可达兴趣）。
  3. 遍历这些兴趣节点的邻居，过滤出以 `activity:` 为前缀的节点（2跳可达活动）。
  4. 过滤掉学生已经直接报名（已存在 `student -- activity` 边）的活动。
  5. 将活动去除前缀，按字典序排序后返回。

### 2. 两跳搭子推荐与 Jaccard 相似度排序 (recommend_buddies)
* **路径语义**：`Student -> Interest -> Other Student`（2跳）。
* **基础推荐**：通过查找与自己拥有共同兴趣的同学（排除了自己），得到所有搭子列表。
* **Jaccard 相似度排序 (扩展功能)**：
  为了提供更高质的匹配，计算每个搭子与当前学生的 Jaccard 相似度并进行降序排列：
  $$J(A, B) = \frac{|Interests_A \cap Interests_B|}{|Interests_A \cup Interests_B|}$$
  * **社交特权模式**：若匹配对象开启了社交特权（Social Boost），其 Jaccard 相似度会乘以 `1.3` 倍，上限不超过 `1.0`。

### 3. BFS 最短路径查找 (find_path)
实现任意两学生节点之间的最短路径，用以展示“六度分割”理论。
* **路径结构**：`Student A -> Interest/Activity -> Student B`
* **隐私过滤**：在 BFS 队列扩展时，系统会检查下一个节点的隐私状态（`isPrivate`）。如果是中间学生节点且开启了隐私模式，则跳过该节点（绕路）；如果目标学生节点开启了隐私，直接返回无路径，保护学生社交隐私。

### 4. 连通社群发现 (connected_components)
* **算法逻辑**：图的极大连通子图划分。
* **实现步骤**：
  1. 初始化 `visited` 集合。
  2. 遍历图的所有节点。若当前节点未访问，则以该节点为起点启动一次 BFS。
  3. BFS 遍历时，将遍历到的所有节点放入当前 Component 列表，并标记为已访问。
  4. 重复上述步骤直至所有节点访问完毕，返回所有 Component 列表。

## 📈 五、核心操作复杂度分析

设 $V$ 为图的节点总数，$E$ 为图的边总数，$d_{\text{avg}}$ 为图顶点的平均度数，$B$ 为与当前学生有共同兴趣的搭子数量。

| 核心操作 | 时间复杂度 | 空间复杂度 | 算法描述 / 备注 |
| :--- | :--- | :--- | :--- |
| **构建图 (Build Graph)** | $O(E)$ | $O(V + E)$ | 遍历 CSV 数据，每次 `add_edge` 为 $O(1)$ 的哈希/Set操作。 |
| **打印邻接表 (Print Graph)** | $O(V \log V + E)$ | $O(1)$ 辅助 | 遍历并排序所有 Key，打印每个顶点的邻居。 |
| **活动推荐 (recommend_activities)** | $O(d_{\text{avg}}^2)$ | $O(d_{\text{avg}}^2)$ | 遍历兴趣分支（最多 $d_{\text{avg}}$）并遍历活动邻居。因图高度稀疏，度数通常 $\le 10$，速度极快。 |
| **搭子推荐 (recommend_buddies)** | $O(d_{\text{avg}}^2)$ | $O(B)$ | 路径为 `Student -> Interest -> Student`。 |
| **连通分量 (connected_components)**| $O(V + E)$ | $O(V)$ | 标准的 BFS 全图遍历，每个顶点和边均只入队/被处理一次。 |
| **Jaccard 搭子排序 (Ranked Buddies)**| $O(B \cdot d_{\text{avg}})$ | $O(B)$ | 对 $B$ 个搭子，分别计算其与当前学生兴趣的交集和并集（集合大小 $\le d_{\text{avg}}$）。 |
| **BFS 最短寻路 (find_path)** | $O(V + E)$ | $O(V)$ | 使用哈希 Map 记录前驱节点，寻路终止时反向回溯。空间复杂度比存储整条路径 of BFS 更优。 |

## 🧪 六、边界情况与测试用例设计

本系统已通过 Pytest 后端测试（21 个用例）和 Vitest 前端测试（26 个用例），覆盖了以下边界：
1. **重复添加同一条边**：邻接表采用 `defaultdict(set)`，底层是哈希集合（Set），天然具备幂等性，重复添加不会增加冗余边。
2. **学生存在但无任何兴趣**：邻接表中该学生节点的邻居集合为空，两跳推荐算法直接返回空列表 `[]`，不会报错或崩溃。
3. **查询不存在的实体**：程序中有显式的 `key in self.graph` 存在性校验，不满足时安全返回空值或 `None`。
4. **命名冲突测试**：学生、活动和兴趣节点具有重名（如学生“篮球”、活动“篮球”和兴趣“篮球”），在图中被正确标识为 `student:篮球`、`activity:篮球`、`interest:篮球`，相互独立互不干扰。

## 🤖 七、AI Coding 结对编程过程反思

在开发 `Campus Buddy` 的过程中，人机结对协作分工明确，效率得到了极大的提升：
* **规则约束**：在开始编码前，向 AI 明确了“无向无权图”与“带类型前缀的邻接表结构”这两个刚性限制，促使 AI 生成了基于 `defaultdict(set)` 的高可读性核心图类。
* **漏洞审查**：在第一版寻路代码中，AI 倾向于在 BFS 队列中直接存储累积的 Path 数组，这在稠密图下会导致巨大的空间开销。经过人工代码审查后，指示 AI 改为“利用 Parent-Map 记录父节点，最后逆序回溯”的经典方案，将 BFS 空间复杂度控制在严格的 $O(V)$。
* **逻辑修正**：在 Jaccard 相似度计算中，AI 忽略了当并集为空（即没有任何兴趣关联）时可能会出现除零错误的问题。通过人工提示，AI 在分母处增加了 `if not union: jaccard = 0.0` 的安全分支。
* **解耦重构**：当项目规模变大时，AI 辅助将 Canvas 渲染与数据服务进行解耦，实现了面向对象的 Painter 架构分离，保证了可视化代码的维护性。
